SQL Execution Order Visualizer

SQL doesn't run in the order you write it. You write SELECT first, but it executes almost last. Understanding this order explains why you can't use a WHERE alias, why HAVING exists, and how the database stays fast.

  1. Query planner — Before a single row is touched, the database reads your entire query and builds an execution plan. It picks which indexes to use and the cheapest join order.
  2. FROM & JOIN — The database fetches your tables and combines them. CTEs and subqueries run at this point too. The result is a big, unfiltered working dataset.
  3. ON condition — The ON clause filters which row pairs from the join survive. Only matching pairs move forward.
  4. OUTER JOIN — If you used LEFT, RIGHT, or FULL OUTER JOIN, unmatched rows are added back into the result here, with NULLs filling the columns from the missing side.
  5. WHERE clause — Rows that don't match the condition are removed. Because this runs before SELECT, you cannot reference column aliases defined in SELECT.
  6. GROUP BY — Rows sharing the same GROUP BY value(s) are collapsed into one group each. After this step, individual row values are gone.
  7. Aggregate functions — SUM, COUNT, AVG, MIN, MAX are calculated across each group. One output row is produced per group.
  8. HAVING clause — HAVING filters groups, just like WHERE filters rows. It runs after aggregation, so you can write conditions like HAVING COUNT(*) > 5.
  9. SELECT — Now the columns you asked for are selected and aliases are assigned. This is step 9, not step 1, which is why aliases can't be used earlier.
  10. DISTINCT — Duplicate rows are removed from the result. It runs after SELECT so it compares the final column values you've selected.
  11. ORDER BY — The result is sorted. Because ORDER BY runs after SELECT, it can reference aliases you created in the SELECT step.
  12. LIMIT / OFFSET — The final result is trimmed to the requested slice. All prior steps still ran over every qualifying row — LIMIT only cuts at the very end.
